## 一、业务痛点：为什么需要装箱优化

### 精铺模式的装箱困境

亚马逊精铺模式的核心特征是**多SKU小批量**——一个运营管100款SKU，每款首单30件，一批货3000+件散货要发到FBA仓。这与精品模式（1款SKU发1000件）的装箱完全不同：

```
精品模式装箱：1款SKU × 1000件 → 工厂原箱直接发，不需要规划
精铺模式装箱：100款SKU × 30件 → 必须混装，怎么搭配是大学问
```

### 三个核心痛点

| 痛点 | 影响 | 传统解决方式 |
|---|---|---|
| **运费浪费** | 轻货（硅胶/纺织）和重货（金属/陶瓷）不搭配装箱，轻货按体积重计费，白白为"空气"付运费。一批货浪费¥5,000-15,000 | 靠运营经验肉眼判断，效果不稳定 |
| **仓库操作效率低** | 一个箱子里混了30-50款SKU，拣货员要在仓库找30-50个位置取货，每箱耗时15-20分钟，差错率高 | 没有SKU数约束，装箱方案不可控 |
| **成本核算不准** | 装箱方案不记录实际重量和箱型，头程运费只能粗略估算，单SKU利润算不准 | 人工填Excel，容易漏算配置费、抛货损失 |

### 这个技能解决什么

```
输入：SKU数据（编号、重量、尺寸、发货数量）
输出：最优装箱方案（每箱装哪些SKU各几件、用什么箱型、运费多少）

算法自动平衡三个目标：
  ① 运费最低（重轻搭配，减少抛货）
  ② 箱数最少（降低装箱工时和耗材成本）
  ③ 每箱SKU最少（降低仓库拣货难度）
```

## 二、核心优化引擎

### 引擎定位

```
┌───────────────────────────────────────────────────┐
│              optimizer.py（核心优化引擎）            │
│                                                   │
│  输入：SKU数据CSV                                   │
│  算法：贪心初始化（SKU集中）+ 模拟退火               │
│  输出：装箱方案CSV + JSON报告                        │
│                                                   │
│  特点：纯Python标准库，零依赖，任何环境可运行         │
│  规模：处理100+款SKU / 5000+件货，1-2秒出结果       │
└───────────────────────────────────────────────────┘
```

**`scripts/optimizer.py` 是整个技能的核心**。它是一个独立的Python脚本，不依赖任何第三方库（不需要pip install），只要有Python 3.8+就能运行。

这个引擎设计为**工具无关**——可以被多种工具形态调用：

### 引擎的约束策略

| 约束 | 默认值 | 可调参数 | 平衡的现实问题 |
|---|---|---|---|
| 每箱SKU数 | ≤5款 | `--max-sku` | 仓库拣货效率：每多1款SKU，拣货时间+2-3分钟 |
| 单箱重量 | ≤15kg | `--max-wt` | FBA标准：超过15kg收附加费，超过22kg拒收 |
| 填充率 | ≤90% | 代码内 | 留10%给气泡膜/泡沫粒等填充物 |
| 箱型选择 | 1-5号箱 | 代码内 | 排除6-7号箱（体积重16-20kg，装满15kg也抛货）|
| SKU集中 | 惩罚权重0.3 | 代码内 | 同款SKU尽量装到同一个箱子，减少找货位置 |
| 货物覆盖 | 100% | 硬约束 | 所有货物必须装箱，不能遗漏 |

### 算法流程

```
第1步：贪心初始化（SKU集中策略）
  ├── 按SKU分组（同款货的30件视为一个整体）
  ├── 按密度排序（重货SKU先处理）
  ├── 重轻交替（一款重货SKU + 一款轻货SKU 交替装箱）
  └── Best-Fit（放剩余空间最小的箱子，更紧凑）

第2步：模拟退火优化（100,000次迭代）
  ├── 随机扰动1：把A箱的一件货移到B箱
  ├── 随机扰动2：交换两个箱子的各一件货
  ├── 随机扰动3：合并两个箱子
  ├── 接受更好的解（成本下降→立即接受）
  ├── 偶尔接受更差的解（概率接受，跳出局部最优）
  └── 逐渐降温收敛到接近最优

第3步：箱型优化
  └── 对每个箱子选计费重量最小的箱型（1-5号箱自动匹配）

目标函数：min（总计费重量 + SKU数惩罚×权重）
```

## 三、使用方式

### 命令行直接使用

```bash
# 基础用法
python3 scripts/optimizer.py SKU数据.csv

# 自定义参数
python3 scripts/optimizer.py SKU数据.csv --max-sku 10        # 每箱最多10款SKU
python3 scripts/optimizer.py SKU数据.csv --air-price 42      # 空运单价¥42/kg
python3 scripts/optimizer.py SKU数据.csv --output 装箱方案.csv  # 指定输出路径
```

### 输入CSV格式

运营从飞书多维表格导出，或从领星ERP导出后调整：

```
SKU编号,产品名称,品类,单件重量(kg),包装长(cm),包装宽(cm),包装高(cm),发货数量,备注
JP-0001,硅胶沥水垫,硅胶,0.15,30.0,20.0,2.0,30,需抽真空
JP-0002,不锈钢量勺5件套,金属,0.2,15.0,10.0,3.0,30,
```

必填字段：SKU编号、单件重量(kg)、包装长(cm)、包装宽(cm)、包装高(cm)、发货数量

### 输出CSV格式

```
箱号,箱型,外箱尺寸,空运体积重(kg),SKU编号,产品名称,品类,数量,本箱实重(kg),计费重量(kg),是否抛货,超重,头程运费(¥),单件头程(¥)
BOX-001,4号箱(45×40×40),45×40×40,12.0,JP-0009,硅胶冰格,硅胶,23,14.05,14.05,✅实重,✅,632,7.71
```

## 四、工具形态规划

核心引擎 `optimizer.py` 是通用的，后续会对接多种工具形态：

### 形态1：Agent Skill（当前版本 v1.0）

```
运营上传CSV → Agent对话 → 自动执行optimizer.py → 返回结果

支持平台：
  - ZCode（SKILL.md已适配）
  - WorkBuddy（README指导安装）
  - Claude Code / Cursor / 其他AI编程工具（直接调用脚本）
```

### 形态2：飞书智能体（规划中）

```
场景：团队全员使用飞书协同，每个运营的飞书里安装智能体

流程：
  运营在飞书多维表格填写SKU数据
  → 飞书智能体自动读取
  → 调用部署在云端的optimizer.py API
  → 结果自动写回飞书装箱方案表
  → 运营在飞书直接查看方案和仪表盘

技术方案：
  - optimizer.py部署为API（Flask/FastAPI + 云函数）
  - 飞书多维表格自动化触发（HTTP请求节点）
  - 或飞书智能体（Bot）接收指令后调用API
```

### 形态3：飞书多维表格自动化（规划中）

```
场景：运营在飞书多维表格里填好SKU数据，点一下按钮，装箱方案自动生成

流程：
  运营在飞书多维表格"SKU装箱数据表"填写重量/尺寸/发货量
  → 触发自动化（按钮触发 或 定时触发）
  → 飞书HTTP请求节点调用云端API（核心是optimizer.py）
  → API返回装箱方案JSON
  → 飞书自动写入"装箱方案表"（每箱的SKU明细/重量/运费）
  → 仪表盘自动更新（总箱数/总运费/抛货率/箱型分布）
  → 运营在飞书直接查看，无需切换工具

技术方案：
  - optimizer.py 部署为API（Flask/FastAPI，部署到云函数或轻量服务器）
  - 飞书多维表格配置自动化工作流（HTTP请求节点）
  - 数据流：飞书SKU表 → API → 飞书装箱方案表（全自动闭环）
  - 仪表盘：飞书原生仪表盘组件（抛货率/成本对比/箱型分布）
```

### 形态4：Web应用（规划中）

```
场景：独立Web页面，或嵌入更大的系统模块

流程：
  运营浏览器打开网页
  → 上传SKU数据CSV
  → 后端调用optimizer.py
  → 网页展示装箱方案（可视化表格+图表）
  → 下载装箱方案CSV

技术方案：
  - 前端：React/Vue + 表格组件 + ECharts图表
  - 后端：Flask/FastAPI + optimizer.py作为核心模块
  - 部署：云服务器或Docker容器
```

### 形态对比

| 形态 | 运营操作 | 开发量 | 适合阶段 |
|---|---|---|---|
| **Agent Skill**（当前） | 上传CSV+对话 | ✅ 已完成 | Phase 1试点 |
| **飞书智能体** | 飞书对话，发送CSV | 1-2周 | Phase 2团队推广 |
| **飞书多维表格自动化** | 填好数据点按钮，全自动 | 1-2周 | Phase 2团队推广 |
| **Web应用** | 浏览器上传下载 | 2-3周 | Phase 3或整合到其他系统 |

四种形态共享同一个核心引擎 `optimizer.py`，只是调用方式和用户界面不同。

## 五、安装到不同Agent工具

### ZCode

```bash
cp -r packing-optimizer-sa-v1 ~/.zcode/skills/
```

在ZCode对话中说："帮我跑装箱优化"，附上CSV文件。

### WorkBuddy

把文件夹放到WorkBuddy的技能目录。在对话中说：

"用 scripts/optimizer.py 帮我优化这个CSV的装箱方案"

### Claude Code / Cursor / 其他AI工具

不需要安装到技能目录。在对话中说：

"用 packing-optimizer-sa/scripts/optimizer.py 帮我跑装箱优化，输入文件是xxx.csv"

### 纯命令行（不用Agent工具）

```bash
git clone https://github.com/wrangler1024/packing-optimizer-sa.git
cd packing-optimizer-sa
python3 scripts/optimizer.py 你的SKU数据.csv
```
